# Reverse Proxy

Reverse Proxy คือเซิร์ฟเวอร์ที่อยู่หน้าเว็บเซิร์ฟเวอร์ (หรือเซิร์ฟเวอร์แอปพลิเคชัน) และทำหน้าที่เป็นตัวกลางในการส่งต่อคำขอจากไคลเอ็นต์ (เช่น เว็บเบราว์เซอร์ของผู้ใช้) ไปยังเซิร์ฟเวอร์แบ็กเอนด์ที่เหมาะสม และส่งคำตอบกลับมายังไคลเอ็นต์ โดยที่ไคลเอ็นต์ไม่รู้ว่ากำลังติดต่อกับเซิร์ฟเวอร์แบ็กเอนด์โดยตรง

**ประโยชน์ของ Reverse Proxy**

- ซ่อน IP Address ของเซิร์ฟเวอร์จริง: Reverse Proxy จะซ่อน IP address ของเซิร์ฟเวอร์แบ็กเอนด์จากสาธารณะ ทำให้ผู้ไม่หวังดีเข้าถึงเซิร์ฟเวอร์จริงได้ยากขึ้น

- ป้องกันการโจมตี: Reverse Proxy สามารถทำหน้าที่เป็นด่านแรกในการป้องกันการโจมตีบางประเภท เช่น DDoS attacks, SQL injection หรือ Cross-Site Scripting (XSS) โดยการกรองคำขอที่ไม่ถูกต้อง

- การจัดการ SSL/TLS: Reverse Proxy สามารถจัดการการเชื่อมต่อ SSL/TLS ทั้งหมดได้ ทำให้เซิร์ฟเวอร์แบ็กเอนด์ไม่จำเป็นต้องรับภาระนี้ และช่วยให้การจัดการใบรับรองเป็นไปในที่เดียว

- การโหลดบาลานซ์ (Load Balancing): หากมีเซิร์ฟเวอร์แบ็กเอนด์หลายเครื่อง Reverse Proxy สามารถกระจายคำขอขาเข้าอย่างเท่าเทียมกันไปยังเซิร์ฟเวอร์เหล่านั้น เพื่อป้องกันไม่ให้เซิร์ฟเวอร์ใดเซิร์ฟเวอร์หนึ่งทำงานหนักเกินไป และเพิ่มความสามารถในการรองรับผู้ใช้งานพร้อมกัน

- การแคช (Caching): Reverse Proxy สามารถแคชเนื้อหาที่เข้าถึงบ่อยๆ ได้ เมื่อมีคำขอเข้ามาซ้ำอีกครั้ง ก็สามารถส่งเนื้อหาจากแคชได้ทันที โดยไม่ต้องส่งต่อไปยังเซิร์ฟเวอร์แบ็กเอนด์ ช่วยลดภาระงานของเซิร์ฟเวอร์แบ็กเอนด์และลดเวลาในการตอบสนอง

- โฮสต์หลายเว็บไซต์บน IP เดียวกัน: Reverse Proxy ช่วยให้คุณสามารถโฮสต์เว็บไซต์หรือแอปพลิเคชันหลายตัวบนเซิร์ฟเวอร์แบ็กเอนด์ที่แตกต่างกันได้ โดยใช้ IP address สาธารณะเพียงหนึ่งเดียว โดยการกำหนดเส้นทางคำขอตามชื่อโดเมน (virtual hosting)

- ลดการเปิดเผยพอร์ต: แทนที่จะเปิดพอร์ตของเซิร์ฟเวอร์แบ็กเอนด์หลายๆ พอร์ตออกสู่ภายนอก Reverse Proxy จะเปิดเพียงพอร์ตเดียว (เช่น 80 สำหรับ HTTP และ 443 สำหรับ HTTPS) แล้วส่งต่อคำขอภายใน

- การบำรุงรักษาโดยไม่มีผลกระทบ: คุณสามารถบำรุงรักษาหรืออัปเดตเซิร์ฟเวอร์แบ็กเอนด์บางเครื่องได้โดยไม่ต้องหยุดบริการทั้งหมด โดยการกำหนดให้ Reverse Proxy ส่งต่อคำขอไปยังเซิร์ฟเวอร์เครื่องอื่น

- การเขียน URL ใหม่ (URL Rewriting): สามารถปรับเปลี่ยนหรือเขียน URL ใหม่ได้ก่อนที่จะส่งคำขอไปยังเซิร์ฟเวอร์แบ็กเอนด์

**ตัวอย่างการใช้งาน**

สมมติว่าคุณมีเว็บแอปพลิเคชัน 3 ตัวที่รันอยู่บนพอร์ตที่ต่างกันบนเซิร์ฟเวอร์ภายในของคุณ:

- แอปพลิเคชัน A: http://localhost:3000 (Node.js)

- แอปพลิเคชัน B: http://localhost:5000 (Python Flask)

- แอปพลิเคชัน C: http://localhost:8000 (PHP Apache)

คุณต้องการให้ผู้ใช้งานเข้าถึงแอปพลิเคชันเหล่านี้ผ่านโดเมนที่แตกต่างกัน โดยใช้ Nginx Proxy Manager:

- app-a.yourdomain.com -> http://localhost:3000

- app-b.yourdomain.com -> http://localhost:5000

- app-c.yourdomain.com -> http://localhost:8000

ด้วย Nginx Proxy Manager คุณสามารถตั้งค่า "Proxy Host" สำหรับแต่ละโดเมน ชี้ไปยัง IP address ภายในและพอร์ตที่ถูกต้องได้อย่างง่ายดาย และยังสามารถขอ Let's Encrypt SSL certificate สำหรับแต่ละโดเมนได้โดยอัตโนมัติ ทำให้การเชื่อมต่อเป็น HTTPS
